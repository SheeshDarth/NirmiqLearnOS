# NirmiqLearn OS — Clean Room Architecture Standard

**No code enters this directory unless it passes every rule in this document.**
All rules are grounded in the actual codebase (`CODEBASE_MAP.md`, `CLAUDE.md`).
This is not aspirational — it is a gate.

---

## 0. What the Clean Room Is

This directory is where new services, components, and utilities are drafted
before they are allowed into the main tree. A piece of code graduates when:

1. It passes every rule in this document (self-checklist below).
2. `npm run lint && npm run typecheck && npm run build` all pass with zero warnings.
3. A second reviewer (human or `/council-review`) has signed off on any new
   abstraction, dependency, or DB schema change.

Until those three gates clear, code lives here and **nowhere else**.

## 1. File Structure Rules

### 1.1 Where every file type lives — non-negotiable

```
lib/services/<domain>.service.ts     <- all DB access + business logic
lib/validators/<domain>.schema.ts    <- Zod schemas for that domain's forms
lib/utils.ts                         <- client-safe pure helpers (no Node imports)
lib/utils/server.ts                  <- server-only FormData helpers
lib/types.ts                         <- shared result type only
lib/db/schema.ts                     <- Drizzle table definitions
lib/db/client.ts                     <- single db instance export
components/<domain>/                 <- React components grouped by feature
app/(app)/<page>/page.tsx            <- Server Component entry point
app/(app)/<page>/actions.ts          <- Server Actions for that page
app/(app)/workspaces/[id]/<feature>/ <- workspace-scoped feature pages
```

**Violations that will be rejected:**
- A second service file for an existing domain (e.g. `workspace-v2.service.ts`).
- A `utils/helpers.ts` or `utils/common.ts` catch-all.
- Any `.tsx` file directly in `app/` that is not a `layout.tsx`, `page.tsx`,
  `error.tsx`, or `loading.tsx`.
- A `lib/store/` or `lib/context/` directory (there is no global client state).
- A `types/` directory (shared types live in `lib/types.ts` only).

### 1.2 Naming conventions

| Thing | Convention | Example |
|---|---|---|
| Service file | `<domain>.service.ts` | `concept-link.service.ts` |
| Validator file | `<domain>.schema.ts` | `concept-link.schema.ts` |
| Component file | `PascalCase.tsx` | `ConceptLinkCard.tsx` |
| Server Action file | `actions.ts` | co-located with its page |
| DB migration | generated by `npm run db:generate` only | never hand-named |
| Script (throwaway) | `_<name>.mts` | deleted after use, never committed |

### 1.3 File size limit

No single file may exceed **400 lines**. If it does, the file is doing more
than one thing. Split along domain boundaries, not along line counts.

### 1.4 One export per responsibility

A service file exports functions, not classes or singleton objects.
A component file has exactly **one default export** — the component.
A validator file exports exactly **one schema per form intent** plus the
inferred `Input` type.

## 2. Type Definitions

### 2.1 The `ServiceResult<T>` contract — mandatory for every service function

Every async service function **must** return `ServiceResult<T>`:

```ts
// lib/types.ts — do not copy this, import it
type ServiceResult<T> =
  | { ok: true; data: T }
  | { ok: false; error: string; code?: string };
```

**Rules:**
- Never `throw` across service boundaries. Catch inside the service, return
  `{ ok: false, error: "...", code: "DB_ERROR" }`.
- Callers must always branch on `result.ok`. No `result.data` without the check.
- `code` is optional but must be a short ALL_CAPS string when provided
  (`"NOT_FOUND"`, `"DB_ERROR"`, `"VALIDATION_ERROR"`).

```ts
// CORRECT
export async function getWorkspaceById(id: string): Promise<ServiceResult<Workspace>> {
  try {
    const [row] = await db.select().from(workspaces).where(eq(workspaces.id, id)).limit(1);
    if (!row) return { ok: false, error: "Workspace not found", code: "NOT_FOUND" };
    return { ok: true, data: row };
  } catch {
    return { ok: false, error: "Failed to fetch workspace", code: "DB_ERROR" };
  }
}

// WRONG — never do this
export async function getWorkspaceById(id: string): Promise<Workspace> {
  const [row] = await db.select()...;
  if (!row) throw new Error("Not found"); // throws across layer boundary
  return row;
}
```

### 2.2 DB row types — infer from schema, never hand-write

```ts
// CORRECT
import { workspaces } from "@/lib/db/schema";
export type Workspace = typeof workspaces.$inferSelect;

// WRONG — structural mismatch risk when schema changes
export type Workspace = {
  id: string;
  title: string;
  // ...hand-written fields
};
```

Derived/presentation types (with parsed JSON columns) must extend the inferred type:

```ts
export type LearningMap = Omit<RawLearningMap, "modulesJson" | "checkpointsJson"> & {
  modules: LearningModule[];
  checkpoints: Checkpoint[];
};
```

### 2.3 Zod schemas — one per form intent, Zod v4 syntax

```ts
import { z } from "zod";

export const createWorkspaceSchema = z.object({
  title: z.string().min(1).max(200),
  type: z.enum(["project", "dsa", "exam", "topic"]),
  description: z.string().max(500).optional(),
  goal: z.string().max(1000).optional(),
});

export type CreateWorkspaceInput = z.infer<typeof createWorkspaceSchema>;
```

**Rules:**
- Schema lives in `lib/validators/<domain>.schema.ts`.
- Actions call `.safeParse()` — never `.parse()` (it throws).
- Services import only the inferred `Input` type, not the schema.
- No inline Zod inside a service or action file.

### 2.4 Client/server boundary — `import type` rule

Client components may not import runtime values from server modules.

```ts
// CORRECT — in a client component
import type { Workspace } from "@/lib/services/workspace.service";

// WRONG — db is a Node.js native module that crashes the client bundle
import { db } from "@/lib/db/client";
```

Test: if a file has `"use client"`, every import must be either (a) a
type-only import from a server module, or (b) from a module safe to bundle
in the browser.

### 2.5 No `any` — strict TypeScript required

```ts
// WRONG
const data: any = JSON.parse(raw);

// CORRECT
const data: unknown = JSON.parse(raw);
if (!Array.isArray(data)) return [];
return data as string[];
```

## 3. State Management

### 3.1 The fundamental rule: reads are Server Components, mutations are Server Actions

There is no client-side data fetching in this application.
No `useEffect(() => { fetch(...) }, [])`. No SWR. No React Query.

```
DB -> Drizzle -> Service function -> Server Component -> rendered HTML
                                     (page.tsx, async)

<form action={serverAction}> -> Server Action -> Service -> DB -> revalidatePath
```

### 3.2 When to use `"use client"`

Add `"use client"` **only** when a component needs one of:

| Need | Example |
|---|---|
| User event handlers | `onClick`, `onChange`, `onSubmit` |
| Browser-only APIs | `localStorage`, `window`, `navigator` |
| Refs | `useRef` for focus, scroll, canvas |
| Controlled form state | `useState` for input values before submit |
| Third-party browser lib | `force-graph`, `three` (must also use `next/dynamic`) |

Presentation with no interaction = Server Component.

### 3.3 `useActionState` — the only pattern for form mutations

```tsx
"use client";
import { useActionState } from "react";
import { myAction, type MyState } from "./actions";

const INITIAL: MyState = { status: "idle" };

export function MyForm() {
  const [state, action, pending] = useActionState(myAction, INITIAL);
  return (
    <form action={action}>
      {/* fields */}
      <button disabled={pending}>Submit</button>
    </form>
  );
}
```

- `pending` from `useActionState` is the loading state — no extra `useState`.
- The action state type is a discriminated union on `status`.
- Never mutate state on the client. The client reflects DB state after `revalidatePath`.

### 3.4 Local `useState` — only for ephemeral UI state

```ts
// ALLOWED — transient UI, no persistence needed
const [collapsed, setCollapsed] = useState(false);

// NOT ALLOWED — this should live in the DB
const [questions, setQuestions] = useState<Question[]>([]);
```

If state needs to survive a page refresh → DB.
If state needs to be shared → pass as props from a Server Component.

### 3.5 No global client state — ever

No Zustand, Redux, Jotai, Recoil, or React Context for data.
Context is allowed only for pure UI concerns after a `/council-review` sign-off.

## 4. Performance Standards

### 4.1 `force-dynamic` — mandatory on DB-reading pages

```ts
export const dynamic = "force-dynamic";
```

Must be the first export in any `page.tsx` that calls a service function.
Without it, Next.js pre-renders with stale data and never re-fetches.

### 4.2 Parallel DB queries with `Promise.all`

```ts
// CORRECT
const [wsResult, mapResult, linksResult] = await Promise.all([
  getWorkspaceById(id),
  getLearningMapByWorkspaceId(id),
  getConceptLinksByWorkspaceId(id),
]);

// WRONG — waterfall, 3x slower
const wsResult = await getWorkspaceById(id);
const mapResult = await getLearningMapByWorkspaceId(id);
const linksResult = await getConceptLinksByWorkspaceId(id);
```

### 4.3 Heavy browser-only libraries — `next/dynamic + ssr: false` required

```ts
// components/<domain>/<LibName>Loader.tsx
import dynamic from "next/dynamic";

const LibComponent = dynamic(() => import("./<LibName>"), { ssr: false });

export default function LibLoader(props: ...) {
  return <LibComponent {...props} />;
}
```

The `Loader` file is what pages import. The component that imports the
heavy library is **never** imported directly from a server-reachable module.

Reference: `components/learning-map/KnowledgeGraphLoader.tsx` wrapping
`KnowledgeGraph.tsx` (force-graph / three).

Violation consequence: Turbopack worker crash —
"Jest worker encountered N child process exceptions" on dynamic routes.

### 4.4 Native Node modules — `serverExternalPackages` required

```ts
// next.config.ts
serverExternalPackages: ["better-sqlite3"],
```

Add any new native module here before using it.

### 4.5 No prop drilling past two levels

If a value passes through more than two component layers, restructure:
fetch it at the Server Component page level and pass directly to the leaf.
Never solve this with a store.

## 5. DB / Migration Rules

### 5.1 Schema changes — generated, never hand-written

```sh
# Stop the dev server first (it holds a write lock on the DB file)
npm run db:generate   # drizzle-kit assigns a correct journal timestamp
npm run db:migrate    # applies pending migrations
```

Never hand-write a migration's `when` timestamp in `_journal.json`.
An out-of-order timestamp causes Drizzle to silently skip the migration.

### 5.2 Cascading deletes — always set `onDelete` on foreign keys

```ts
workspaceId: text("workspace_id")
  .notNull()
  .references(() => workspaces.id, { onDelete: "cascade" }),
```

Optional foreign keys use `onDelete: "set null"`.

### 5.3 JSON columns — named `Json`, parsed in the service only

```ts
// schema
modulesJson: text("modules_json").notNull().default("[]"),

// service — always parse before returning to callers
function toPresentation(raw: RawMap): Map {
  return { ...raw, modules: JSON.parse(raw.modulesJson) as Module[] };
}
```

Never expose raw JSON strings to components.

### 5.4 `db` import — one correct path only

```ts
import { db } from "@/lib/db/client";   // CORRECT
import { db } from "@/lib/db";           // WRONG
```

## 6. Component Rules

### 6.1 Server Component checklist

- `async` function with `await` on service calls
- `export const dynamic = "force-dynamic"` at the top
- No `"use client"` directive
- No `useState`, `useEffect`, `useRef`
- Calls `notFound()` from `next/navigation` when resource is missing
- Props are plain serialisable values only

### 6.2 Client Component checklist

- `"use client"` as the very first line (before all imports)
- Server module imports are `import type` only
- No direct DB or service imports
- `useActionState` for any mutation that touches the DB
- `suppressHydrationWarning` on shell form controls a browser extension might annotate

### 6.3 Icon rule — lucide-react only, verify before use

```ts
import { Map, ArrowRight, Code2 } from "lucide-react";
```

There is no `Github` icon — use `GitBranch`.
Before using a new icon, confirm the export exists in the installed version.

### 6.4 Styling — the established palette only

| Role | Token |
|---|---|
| Surface (primary) | `bg-[#0d1117]` |
| Surface (deep) | `bg-[#0a0c10]` |
| Border | `border-zinc-800` / hover `border-zinc-700` |
| Body text | `text-zinc-100` / `zinc-300` / `zinc-400` |
| Muted | `text-zinc-500` / `zinc-600` |
| Primary accent | `cyan-400` / `cyan-500` |
| DSA / concepts | `violet-400` |
| Success | `emerald-400` |
| Warning | `amber-400` |
| Error | `red-400` |

No new colours. No inline `style=` except for dynamic numeric values
(e.g., `style={{ width: `${pct}%` }}`).

Card pattern — reuse, do not reinvent:
```
bg-[#0d1117] border border-zinc-800 rounded-lg
```

## 7. Server Action Rules

### 7.1 Required structure — every action must follow this exactly

```ts
"use server";

import { mySchema } from "@/lib/validators/my-domain.schema";
import { myServiceFn } from "@/lib/services/my-domain.service";
import { revalidatePath } from "next/cache";
import { getString, getUUID } from "@/lib/utils/server";

export type MyActionState =
  | { status: "idle" }
  | { status: "error"; message: string }
  | { status: "success" };

export async function myAction(
  _prev: MyActionState,
  formData: FormData
): Promise<MyActionState> {
  const workspaceId = getUUID(formData, "workspaceId");
  const title = getString(formData, "title");

  const parsed = mySchema.safeParse({ workspaceId, title });
  if (!parsed.success) {
    return { status: "error", message: parsed.error.issues[0]?.message ?? "Validation error" };
  }

  const result = await myServiceFn(parsed.data);
  if (!result.ok) return { status: "error", message: result.error };

  revalidatePath("/workspaces/[id]/my-feature", "page");
  return { status: "success" };
}
```

- Always `getString` / `getUUID` — never `formData.get("x") as string`.
- Always `.safeParse()` — never `.parse()`.
- Always `revalidatePath` after a successful write.
- Never `redirect()` inside a `try/catch`.

## 8. What Is Explicitly Banned

| Pattern | Reason |
|---|---|
| `formData.get("field") as string` | Use `getString()`; skips trim and null checks |
| `throw new Error(...)` in a service | Use `ServiceResult`; throwing crosses the layer boundary |
| `import { db } from "@/lib/db"` | Wrong path; use `@/lib/db/client` |
| `useEffect(() => { fetch(...) }, [])` | Use a Server Component |
| Any global store | No Zustand / Redux / Jotai in this project |
| `any` type | Use `unknown` + narrowing |
| Hand-writing a migration `when` field | Use `npm run db:generate` |
| Importing `force-graph` / `three` outside a `next/dynamic` wrapper | Turbopack worker crash |
| `better-sqlite3` without `serverExternalPackages` | Same crash |
| A new colour outside the established palette | Use existing tokens |
| A new npm dependency without `/council-review` | Justify the need first |
| Comments explaining what the code does | Name identifiers well; comment only WHY |
| Multi-paragraph docstrings | One-line max, only for non-obvious WHY |
